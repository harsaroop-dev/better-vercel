# 🚀 Better-Vercel: Custom CI/CD Engine & Static Hosting Platform

A high-performance, automated deployment pipeline built from scratch. This project replicates the core CI/CD and static-hosting infrastructure of platforms like Vercel, capable of catching GitHub webhooks, orchestrating isolated Docker builds, and deploying compiled frontend applications (React, Vite, Static HTML) directly to AWS S3 with wildcard DNS routing.

## ✨ Core Architecture & Features

- **Automated Webhook Pipeline:** Listens for `push` events on GitHub, automatically pulling source code into the deployment queue.
- **Isolated Build Sandboxes:** Provisions temporary, secure Docker containers (`node:20-slim`) to compile applications without contaminating the host server.
- **Smart Payload Detection:** Intelligently analyzes incoming repositories. Routes React/Vite apps through optimized `npm ci` builds, while bypassing the build step entirely for Static HTML sites.
- **Real-Time Log Streaming:** Pipes standard output from the isolated Docker container directly to the client dashboard via **WebSockets**.
- **Cloud Storage & Routing:** Pushes compiled artifacts (`/dist` or `/build`) to AWS S3, instantly serving them globally via custom wildcard DNS (`*.bettervercel.harsaroop.com`).

## 🧠 Technical Challenges & Hardware Optimization

Building a deployment engine on a 1GB AWS `t2.micro` instance presented severe hardware limitations. I engineered several system-level optimizations to prevent kernel crashes during heavy React/Vite compilations:

### 1. Bypassing the Linux OOM (Out-Of-Memory) Sniper

**The Problem:** Running `npm install` for modern React apps caused massive memory spikes, triggering the Linux Kernel's OOM killer to silently assassinate the build process (`Exit handler never called!`).
**The Solution:** \* Provisioned **2GB of SSD Swap Space** (`/swapfile`) to act as virtual RAM.

- Implemented a strict Node.js memory collar inside the Docker container (`export NODE_OPTIONS="--max-old-space-size=512"`) to force aggressive garbage collection before memory exhaustion.
- Enforced the use of `package-lock.json` and optimized the build script to use `npm ci`, drastically reducing CPU computation and memory overhead.

### 2. Overcoming the Docker Bridge Collapse

**The Problem:** Docker's default virtual network bridge collapsed under the sheer volume of parallel network requests generated by NPM when fetching thousands of dependencies, resulting in silent connection drops.
**The Solution:** Rewired the container initialization to use native host networking (`--network host`), bypassing the fragile virtual bridge and connecting the build sandbox directly to the EC2 network interface.

### 3. File System Locking & Storage Orchestration

**The Problem:** Mounting an external volume directly into Docker for the build step caused translation-layer bottlenecks and file-locking errors (`EACCES` / `ENOENT`) when NPM attempted to write thousands of `node_modules`.
**The Solution:** Re-architected the build script to copy source code into a native, internal container directory (`/build_env`). The heavy I/O operations execute natively, and only the final compiled artifacts are slipped back across the volume bridge to the host server.

## 🗺️ Future Roadmap: Evolution to Serverless

The current engine is highly optimized for **Static/SPA delivery**.
The next architectural evolution involves transitioning from static S3 storage to serverless compute to support **Next.js SSR (Server-Side Rendering)** and API routes. This would require replacing the post-build S3 upload phase with dynamic provisioning of AWS Lambda functions or managed containers (e.g., AWS Fargate) to execute Node.js instances on the fly.

3. Configure .env

   GITHUB_WEBHOOK_SECRET=your_secret
